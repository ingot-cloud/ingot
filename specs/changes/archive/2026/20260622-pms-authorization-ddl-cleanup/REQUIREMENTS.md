# Requirements

## 用户场景

运维/DBA 在应用中心化模型（发布 A）稳定运行一个完整发布周期后，执行一次独立的破坏性数据库清理，使数据库结构与应用中心化模型完全对齐，不再保留旧模型的兼容字段。

## 业务规则

- 仅在满足全部前置条件后执行（见「边界与非目标」）。
- 删除旧字段后，应用、菜单、权限的归属与来源不得出现空值。
- 保留 `platform_role` 布尔列作为角色来源唯一表示（D2）。
- 必须提供可执行的反向迁移脚本与备份恢复方案。
- 必须独立发布，不与其他功能同批上线。

## 边界与非目标

### 前置条件（实施闸门，必须全部满足）

- 应用中心化模型稳定运行至少一个完整发布周期。
- 所有租户均已切换到应用中心化模型。
- 授权差异观测持续为零（发布 A 已移除 shadow，需通过审计/回归确认）。
- 没有旧版本服务实例运行。
- 已完成数据库备份与回滚演练。
- 已确认无外部系统读取待删除字段。

### 非目标

- 不变更任何接口或业务行为。
- 不删除 `platform_role` 布尔列。
- 不重新设计部门数据权限模型。

## 验收标准

- [ ] 代码中不再读取已删除字段（发布 A 已完成 Java 层移除，本次确认无新增引用）。
- [ ] `008_application_authorization_cleanup.sql` 在测试/预发布库成功执行。
- [ ] `rollback_008.sql` 可成功回退新增约束与列删除（结合备份恢复数据）。
- [ ] 数据库不存在 `app_id`、权限来源、节点类型为空的数据。
- [ ] 所有权限可追溯到应用，所有菜单托管权限可追溯到菜单，所有租户应用授权关系唯一。
- [ ] 全新库仅靠 `databases/ingot_core.sql` 可正确启动。
- [ ] 上一正式版本可经 `008` 迁移脚本升级完成。
- [ ] 全量回归、权限安全测试、迁移回归测试通过。
- [ ] 相关运维与 API 文档已同步更新。
