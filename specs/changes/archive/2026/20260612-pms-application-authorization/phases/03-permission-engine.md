# 阶段 3：权限树与鉴权引擎

> 状态：代码已完成，待统一验收。

## 1. 目标

实现基于应用、精确权限和显式通配权限的新鉴权引擎，并以影子模式与旧结果持续比较。

## 2. 权限表达

支持两类授权：

```text
EXACT：contacts:user:create
SUBTREE：contacts:user:*
```

匹配规则：

- 精确权限只能匹配相同编码。
- `contacts:user:*` 匹配 `contacts:user:` 命名空间下的权限。
- `contacts:user:*` 不匹配 `contacts:users:create`。
- `contacts:*` 不匹配其它应用权限。
- 不支持中间通配，例如 `contacts:*:delete`。
- 不支持普通父权限隐式包含子权限。

## 3. 有效权限计算

输入：

- 当前用户。
- 当前租户。
- 用户角色。
- 平台预设角色默认权限。
- 租户追加权限。
- 租户自定义角色权限。
- 应用和租户应用授权状态。

输出：

- 精确权限集合。
- 通配权限集合。
- 可访问应用 ID 集合。
- 权限版本信息。

处理顺序：

1. 获取角色授权。
2. 合并平台默认和租户追加权限。
3. 去重精确权限和通配权限。
4. 过滤禁用权限。
5. 过滤禁用应用。
6. 租户上下文中过滤未授权或过期应用。

不要求在登录时展开所有通配权限为叶子权限，鉴权器可以使用前缀匹配，以避免权限数量增长导致令牌和缓存膨胀。

## 4. 旧权限迁移

现有“父权限拥有全部后代”的绑定需要迁移为显式 `SUBTREE`：

- 对历史角色绑定，判断该绑定是否依赖旧父级继承语义。
- 依赖继承的绑定转换为对应 `:*` 权限。
- 仅需父节点自身的绑定转换为精确权限。
- 无法自动判断时进入人工确认清单。

迁移不能简单地将所有父权限全部转换为通配权限，否则可能扩大授权。

## 5. 鉴权 API

统一提供：

```text
hasPermission(code)
hasAnyPermission(codes)
hasAllPermissions(codes)
getEffectiveAuthorization(userId, tenantId)
```

菜单、按钮和后端接口必须使用同一个权限匹配器。

## 6. 缓存

缓存内容：

- 用户角色引用。
- 角色授权集合。
- 租户有效应用集合。
- 用户有效授权快照。

失效事件：

- 应用状态变化。
- 权限新增、修改、禁用或删除。
- 租户应用授权变化。
- 角色权限变化。
- 用户角色变化。

必须在事务提交后失效缓存。

## 7. 影子验证

`shadow` 模式下：

- 线上仍返回旧权限结果。
- 同步或异步计算新结果。
- 对精确权限、通配展开结果和应用范围进行比较。
- 差异按预期迁移差异和异常差异分类。

## 8. 安全与审计

以下操作必须记录审计：

- 应用级 `app:*` 授权。
- 模块级 `module:*` 授权。
- 通配权限撤销。
- 权限编码迁移。
- 权限跨父节点移动。

## 9. 验收标准

- 精确和通配权限匹配测试全部通过。
- 不存在跨应用通配匹配。
- 普通父权限不再隐式包含子权限。
- 新旧模型差异可以定位到用户、租户、角色和权限。
- 所有历史父级授权都有明确迁移结果。
- 影子模式不影响原授权接口返回。
- 权限变更后缓存能够在事务提交后失效。
- 性能测试中单次权限判断无数据库查询。

## 10. 退出条件

- 预发布环境新旧权限结果的非预期差异为零。
- 生产影子模式达到约定观察周期，且非预期差异率为零。

## 11. 回滚

将配置切回 `legacy`。新权限字段和迁移数据继续保留，停止使用新鉴权器。
