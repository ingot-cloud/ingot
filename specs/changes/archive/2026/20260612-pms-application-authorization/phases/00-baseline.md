# 阶段 0：基线与迁移护栏

> 状态：代码已完成，待统一验收。

## 1. 目标

在不改变线上行为的前提下，固定现有菜单、权限、角色和租户应用授权结果，为后续双模型迁移提供可比较基线。

## 2. 实现内容

### 2.1 数据审计

提供只读审计脚本或管理任务，检查：

- 菜单是否存在无效父节点。
- 菜单与自动权限是否一一对应。
- 菜单权限父级是否与菜单父级一致。
- 应用的根菜单和根权限是否存在。
- 权限编码是否重复。
- 角色权限是否引用已删除权限。
- 租户私有关系是否缺少租户、角色或权限。
- `platform_role` 布尔标记与实际角色来源是否一致。

审计结果输出总数、异常明细和可自动修复建议，不在本阶段自动修改业务数据。

### 2.2 行为基线测试

为以下现有行为增加集成测试：

- 用户角色查询。
- 平台角色与租户私有权限并集。
- 不可用应用权限过滤。
- 菜单权限过滤和祖先菜单补齐。
- 平台菜单与组织菜单隔离。
- 租户管理员初始化。

### 2.3 差异观测框架

增加可配置的影子计算入口：

```text
authorization.model = legacy | shadow | application
```

- `legacy`：只使用旧模型。
- `shadow`：旧模型返回结果，新模型仅计算并记录差异。
- `application`：使用新模型返回结果。

差异日志至少包含：

- tenantId
- userId
- legacy permission codes
- application permission codes
- legacy menu IDs
- application menu IDs
- 缺失项和新增项

不得记录密码、令牌或其它敏感信息。

### 2.4 数据快照

迁移前生成以下逻辑快照：

- 应用及对应菜单、权限范围。
- 每个租户的实际有效应用。
- 平台预设角色默认权限。
- 租户追加权限和自定义角色权限。
- 测试用户的有效权限与菜单。

## 3. 测试场景

- 无角色用户。
- 仅平台角色用户。
- 仅租户自定义角色用户。
- 同时拥有平台预设角色和租户角色的用户。
- 租户应用显式启用、显式禁用和无配置场景。
- 父权限授权和叶子权限授权场景。
- 已删除或禁用资源场景。

## 4. 验收标准

- 审计工具可以在测试数据库完整运行且不修改数据。
- 所有异常均有明确分类和主键定位信息。
- 现有核心授权流程具备自动化测试。
- `shadow` 开关接入但默认关闭。
- 已选取至少一组平台用户和一组租户用户作为后续迁移对照样本。
- 基线测试在连续三次执行中结果一致。

## 5. 退出条件

- 高风险数据异常已经修复，或形成明确的迁移处理规则。
- 后续阶段可以通过自动化方式比较新旧权限和菜单结果。

## 6. 回滚

本阶段只增加测试、审计和观测代码。关闭影子开关即可停止新增逻辑，不涉及数据回滚。
