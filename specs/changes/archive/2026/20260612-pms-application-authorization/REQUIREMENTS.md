# 应用中心化授权改造需求

## 1. 背景与目标

当前 PMS 将租户应用与根菜单、根权限一一绑定（`platform_app.menu_id`、`platform_app.permission_id`），通过 `org_type` 区分平台菜单/权限与租户菜单/权限，父权限绑定后隐式包含全部子权限。该模型在应用扩展、权限精确控制和租户套餐授权方面存在边界模糊问题。

本次改造目标：将**应用**提升为菜单和权限的统一资源归属与租户授权边界，同时保留菜单导航、权限能力和角色授权各自清晰的职责。

## 2. 用户场景

### 2.1 平台管理员维护应用资源

**触发**：平台管理员在应用管理中创建或维护租户应用（如「通讯录」）。

**期望**：

- 在应用详情中统一管理基本信息、菜单树和操作权限树。
- 创建应用时自动生成 `app_code:*` 根权限。
- 创建菜单时自动创建托管导航权限，名称和父级与菜单同步。
- 应用存在租户授权或角色绑定时不可物理删除，只能禁用。

### 2.2 平台管理员配置平台后台

**触发**：平台管理员维护平台后台模块（如「平台配置」「组织管理」）。

**期望**：

- 平台模块作为 `PLATFORM` 应用管理，不授权给租户。
- 平台用户菜单只包含 `PLATFORM` 应用下的菜单。
- 根菜单按应用 `sort` 排序，同一应用内再按菜单 `sort` 排序。

### 2.3 租户管理员开通应用

**触发**：平台为租户开通或停用某个 `TENANT` 应用。

**期望**：

- 未显式授权的应用默认不可用（切换前需迁移现有继承逻辑，避免权限突降）。
- 可记录授权来源（套餐、人工、系统初始化）和有效期。
- 应用禁用或过期后，相关权限立即失效，但历史角色绑定保留。

### 2.4 租户管理员配置角色

**触发**：租户管理员为预设角色追加权限，或创建/编辑自定义角色。

**期望**：

- 平台预设角色的默认权限显示为锁定，租户不可取消。
- 租户只能追加或撤销自己追加的权限。
- 自定义角色只能选择租户已授权应用范围内的权限。
- 支持绑定应用根通配（`contacts:*`）、模块通配（`contacts:user:*`）或精确权限（`contacts:user:create`）。
- 通绑通配权限时，管理界面明确提示其动态范围，并写入审计日志。

### 2.5 普通用户登录使用系统

**触发**：平台用户或租户用户登录后加载菜单和权限。

**期望**：

- 前端继续接收原有菜单树结构，不增加应用层导航节点。
- `OPEN` 菜单跳过角色权限检查，但不跳过应用状态和租户应用授权检查。
- `PERMISSION` 菜单需具备对应导航权限。
- 按钮显隐和后端 API 鉴权使用同一 `ACTION` 权限编码。
- 精确权限只匹配自身；显式 `:*` 权限动态包含命名空间内后代。

### 2.6 租户管理员（系统管理员角色）

**触发**：受保护的系统管理员角色用户访问租户后台。

**期望**：

- 动态拥有当前租户全部有效应用的 `app_code:*` 根权限。
- 新开通应用或应用新增权限后，管理员无需批量补数据即可访问。
- 动态策略仅对受保护的系统管理员角色生效，不根据角色名称判断。

## 3. 业务规则

| 优先级 | 规则 |
|---|---|
| P0 | 所有菜单和权限必须归属于应用（`app_id` 非空）。 |
| P0 | 应用区分 `PLATFORM` 与 `TENANT`；`TENANT` 应用才可授权给租户。 |
| P0 | 精确权限只匹配自身；显式 `:*` 权限动态包含后代；禁止普通父权限隐式包含子权限。 |
| P0 | 租户只能访问和分配已授权应用中的权限。 |
| P0 | 平台预设角色默认权限租户不可削减，只能追加。 |
| P1 | 菜单权限由菜单生命周期自动维护（创建/改名/移动/删除）。 |
| P1 | 操作权限在应用权限树中统一维护，菜单托管权限在权限管理中只读。 |
| P1 | 一个应用可包含多个根菜单；应用本身不作为菜单节点返回。 |
| P1 | 根菜单顺序：`app.sort` → `menu.sort` → `menu.id`；子菜单在父节点下按 `menu.sort` → `menu.id`。 |
| P1 | 应用 `code` 创建后不可通过普通更新接口修改，同时作为权限命名空间。 |
| P2 | 子菜单不能跨应用移动；子权限编码必须位于父权限命名空间内。 |
| P2 | 通配授权、权限编码迁移、跨父节点移动必须审计。 |

## 4. 边界与非目标

### 4.1 兼容与迁移约束

- 采用兼容迁移，不一次性替换线上模型。
- 阶段 0–5 必须保留配置级回滚能力。
- 旧字段在阶段 5 之前仍是部分读路径的事实来源；新增写操作需双写新旧字段。
- 切换租户应用默认策略（继承全局 → 默认拒绝）前，必须为活跃租户生成等价授权记录。

### 4.2 异常与边界场景

- 应用全局禁用优先于租户授权启用状态。
- 菜单祖先目录被禁用时，后代菜单不可见。
- 平台默认权限若超出租户应用授权范围，暂不生效但仍显示来源。
- 同一菜单树出现多个应用归属时，迁移脚本必须停止并输出异常。
- 无法自动判断的历史父级授权绑定，进入人工确认清单，不得批量转为通配权限扩大授权。

### 4.3 非目标

- 不引入完整 ABAC 或策略语言。
- 不把菜单、权限和应用合并为一张表。
- 不重新设计部门数据权限。
- 不强制前端渲染应用一级导航。
- 不为每个 HTTP 接口单独创建权限。
- 不复制平台预设角色到每个租户。

## 5. 验收标准

### 5.1 功能验收

- [ ] 每个有效菜单、权限均可追溯到唯一应用。
- [ ] 菜单托管权限与菜单一一对应，生命周期同步正确。
- [ ] 精确权限与显式通配权限匹配行为符合 DESIGN 定义。
- [ ] 租户未授权应用不可访问、不可分配。
- [ ] 平台预设角色默认权限不可被租户取消。
- [ ] 菜单树根节点按应用排序，前端协议向后兼容（可附带 `appId`、`appCode`）。
- [ ] 平台用户与租户用户菜单隔离正确。

### 5.2 迁移与兼容验收

- [ ] 阶段 0–5 均可通过配置回滚，且不破坏已回填数据。
- [ ] 影子模式下新旧权限和菜单的非预期差异为零。
- [ ] 租户切换后有效应用数量不意外减少。
- [ ] 所有历史父级授权绑定均有明确迁移结果（自动或人工）。

### 5.3 质量验收

- [ ] 核心授权路径具备自动化集成测试（见 [阶段 0](./phases/00-baseline.md) 测试场景）。
- [ ] 单次权限判断无数据库查询（缓存命中场景）。
- [ ] 权限变更后缓存于事务提交后失效。
- [ ] 通绑通配权限和关键授权变更具备审计记录。

### 5.4 完成验收

- [ ] 全部 7 个阶段验收标准和退出条件满足。
- [ ] 实际数据模型、接口和鉴权行为与 [DESIGN](./DESIGN.md) 一致。
- [ ] `specs/current/pms/application-authorization/` 已根据最终实现更新。
- [ ] 本 change 已记录完成信息并归档。
