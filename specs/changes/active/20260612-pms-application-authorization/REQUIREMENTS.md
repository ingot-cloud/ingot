# 应用中心化授权改造需求

## 1. 目标

将应用提升为菜单和权限的统一资源归属与租户授权边界，同时保留菜单导航、权限能力和角色授权各自清晰的职责。

## 2. 核心需求

- 所有菜单和权限必须归属于应用。
- 应用区分平台应用与租户应用。
- 一个应用可以包含多个根菜单，应用本身不作为菜单节点返回。
- 菜单权限由菜单生命周期自动维护。
- 操作权限在应用权限树中统一维护，并供前端按钮和后端 API 共用。
- 精确权限只匹配自身，显式 `:*` 权限动态包含后代。
- 租户只能访问和分配已授权应用中的权限。
- 平台预设角色不可由租户削减，租户可以追加权限。
- 菜单根节点先按应用排序，再按菜单排序；子菜单在各自父节点下排序。

## 3. 约束

- 采用兼容迁移，不一次性替换线上模型。
- 不把应用、菜单和权限合并为一张表。
- 不重新设计部门数据权限。
- 不强制前端渲染应用一级导航。
- 不为每个 HTTP 接口单独创建权限。
- 不复制平台预设角色到每个租户。

## 4. 验收标准

完整验收标准以 [目标设计](./DESIGN.md) 和 [阶段文档](./phases/) 为准。全部阶段完成前，本变更不得标记为 `completed`。
