# 阶段 5：菜单交付与正式切换

> 状态：待实施。

## 1. 目标

使用应用顺序、菜单顺序和新权限引擎生成最终菜单树，并正式切换授权读路径。

## 2. 菜单生成

平台用户：

- 只查询有效 `PLATFORM` 应用。
- 根据平台角色权限过滤 `PERMISSION` 菜单。

租户用户：

- 只查询当前租户有效授权的 `TENANT` 应用。
- `OPEN` 菜单不检查角色权限。
- `PERMISSION` 菜单检查导航权限。

所有场景都必须：

- 过滤禁用应用。
- 过滤禁用菜单。
- 补齐可见菜单所需的祖先目录。
- 保证祖先目录禁用时后代不可见。

## 3. 排序

根菜单按以下顺序：

```text
app.sort ASC
menu.sort ASC
menu.id ASC
```

每个父菜单的直接子节点按以下顺序：

```text
menu.sort ASC
menu.id ASC
```

实现方式：

1. 获取有序应用列表。
2. 按应用构建菜单树。
3. 递归排序每个父节点的直接子节点。
4. 按应用顺序拼接根节点。

不允许通过把应用伪装为菜单根节点实现排序。

## 4. 返回协议

保持现有菜单树主体结构，新增字段必须向后兼容：

```text
appId
appCode
```

前端当前不需要渲染应用层，但可以使用字段进行调试、未来应用切换或导航分组。

## 5. 正式切换

切换顺序：

1. 管理接口写入切换到新模型。
2. 权限查询切换到新引擎。
3. 菜单查询切换到新生成器。
4. 登录用户权限快照切换到新模型。
5. 关闭旧模型新增写入。

每一步使用独立开关，禁止一次性全部切换。

## 6. 性能验收

重点测量：

- 登录权限加载耗时。
- 菜单树生成耗时。
- 单次权限判断耗时。
- 缓存命中率。
- 应用或角色变更后的缓存失效时间。

菜单生成不能对每个应用、菜单或权限执行独立数据库查询。

## 7. 验收标准

- 根菜单严格按应用排序后再按菜单排序。
- 子菜单在各自父节点下稳定排序。
- 相同排序值使用 ID 保证结果稳定。
- 返回结构不增加应用菜单层。
- 平台用户不会获得租户应用菜单。
- 租户用户不会获得平台应用菜单。
- `OPEN` 和 `PERMISSION` 行为符合定义。
- 新旧模型非预期菜单差异为零。
- 新旧模型非预期权限差异为零。
- 核心接口无越权回归。
- 性能不低于旧模型约定基线。

## 8. 退出条件

- `application` 模式稳定运行一个完整发布周期。
- 线上差异观测无非预期差异。
- 所有客户端均已适配兼容返回字段。

## 9. 回滚

- 权限和菜单读取开关切回 `legacy`。
- 恢复旧写入入口。
- 不回滚新表数据，待修复后重新进行影子比较。
