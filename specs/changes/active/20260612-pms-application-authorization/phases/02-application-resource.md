# 阶段 2：应用中心化资源管理

> 状态：待实施。

## 1. 目标

将应用、菜单和权限的写入入口统一到应用上下文中，建立资源归属约束和菜单托管权限生命周期。

## 2. 管理接口

应用详情提供：

```text
基本信息
菜单管理
权限管理
```

建议接口语义：

```text
POST   /platform/apps
GET    /platform/apps/{appId}
GET    /platform/apps/{appId}/menus
POST   /platform/apps/{appId}/menus
GET    /platform/apps/{appId}/permissions
POST   /platform/apps/{appId}/permissions
```

现有接口保留兼容期，但内部转发到新的应用资源服务。

## 3. 应用生命周期

创建应用时：

1. 校验应用编码唯一。
2. 保存应用。
3. 创建 `app_code:*` 根权限。
4. 在同一事务提交。

删除应用时：

- 存在菜单、操作权限、租户授权或角色绑定时禁止直接删除。
- 提供禁用能力，不提供级联物理删除。

## 4. 菜单生命周期

创建菜单时：

1. 从 URL 上下文取得 `appId`，不信任请求体中的应用 ID。
2. 子菜单校验父菜单属于同一应用。
3. 保存菜单。
4. 创建托管 `NAVIGATION` 权限。
5. 将权限 ID 回写菜单。
6. 同事务提交。

更新菜单时：

- 名称同步到权限。
- 父级调整同步权限父级。
- `access_mode` 只影响鉴权，不删除权限关系。
- 路由变化不自动修改权限编码。

删除菜单时：

- 必须是叶子菜单。
- 托管权限下不能存在操作权限。
- 清理角色权限关联后才允许删除托管权限。

## 5. 操作权限生命周期

操作权限只能在应用内创建：

- 父权限必须属于当前应用。
- `GROUP` 编码必须以 `:*` 结尾。
- `ACTION` 编码必须是精确编码。
- 菜单托管权限在权限接口中只读。
- API 和按钮共用 `ACTION` 权限。

菜单详情可以提供创建操作权限的快捷入口，但必须复用同一个权限领域服务。

## 6. 领域服务边界

建议形成统一应用资源服务，负责：

- 应用归属校验。
- 菜单与权限事务一致性。
- 权限编码规则校验。
- 跨应用移动拦截。
- 缓存失效事件发布。

控制器不得分别拼装菜单和权限写操作。

## 7. 测试场景

- 创建平台应用和租户应用。
- 应用编码重复。
- 应用创建事务回滚。
- 创建多个根菜单。
- 子菜单继承应用。
- 跨应用选择父菜单。
- 菜单权限自动创建。
- `OPEN/PERMISSION` 往返切换。
- 菜单改名和移动。
- 菜单路径变化但权限编码不变。
- 在菜单权限下创建操作权限。
- 直接修改托管权限被拒绝。
- 有租户授权或角色绑定的应用不能删除。

## 8. 验收标准

- 新应用创建后必然存在且仅存在一个应用根权限。
- 菜单和托管权限不存在单边成功状态。
- 所有新增菜单、权限都有正确 `app_id`。
- 跨应用菜单和权限父子关系无法创建。
- `OPEN/PERMISSION` 切换不丢失角色权限关系。
- 权限管理可以完整展示应用权限树。
- 菜单权限只能由菜单生命周期更新。
- 旧管理接口在兼容期内仍可工作。

## 9. 退出条件

- 所有新增资源都通过应用中心化接口创建。
- 旧接口不再产生缺失 `app_id` 的数据。
- 资源写入链路具备自动化事务测试。

## 10. 回滚

- 保留旧接口和旧字段写入。
- 通过功能开关恢复旧管理入口。
- 新创建的应用归属数据不删除，旧代码忽略新增字段。
