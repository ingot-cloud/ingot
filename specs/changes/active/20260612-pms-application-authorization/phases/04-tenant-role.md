# 阶段 4：租户应用与角色授权

> 状态：核心逻辑已完成，灰度策略待验证；待统一验收。

## 1. 目标

正式启用应用级租户授权，并将平台预设角色、租户追加权限和租户自定义角色统一接入新权限表达。

## 2. 租户应用授权

规则：

- 只有 `TENANT` 应用可以授权给租户。
- 没有授权记录默认不可用。
- 授权必须处于启用状态和有效期内。
- 应用全局禁用优先于租户授权。
- 应用失效时保留历史角色绑定，但不参与有效权限计算。

管理能力：

- 开通应用。
- 禁用应用。
- 设置授权来源。
- 设置有效期。
- 查询应用影响的角色和用户。

## 3. 现有租户数据迁移

当前逻辑在没有私有配置时可能继承应用全局状态。切换为默认拒绝前，必须为每个活跃租户生成其当前实际可用应用授权，避免权限突然减少。

迁移结果需要输出：

- 租户数。
- 新建授权数。
- 显式禁用数。
- 无法确定的授权数。

## 4. 预设角色

平台预设角色权限分为：

- 平台默认权限：来自 `platform_role_permission`，租户只读。
- 租户追加权限：来自 `tenant_role_permission_private`，租户可增加和撤销。

授权界面要求：

- 平台默认权限显示锁定状态。
- 租户追加权限正常显示。
- 未授权应用权限不可选择。
- 平台默认权限如果超出租户应用授权范围，则暂不生效，但仍显示其来源。

## 5. 租户自定义角色

租户角色可以选择：

- 应用级通配权限。
- 模块级通配权限。
- 精确权限。

服务端必须校验提交权限全部位于租户有效应用范围内。

## 6. 租户管理员

租户管理员采用动态全应用策略：

```text
当前租户全部有效应用的 app_code:*
```

不为管理员角色保存所有叶子权限。

动态权限必须仅在识别到受保护的系统管理员角色时启用，不能根据角色名称判断。

## 7. 角色关联来源

正式使用：

```text
role_source = PLATFORM | TENANT
```

查询角色时必须同时使用 `role_source + role_id`，禁止只用角色 ID 在两张表中依次查找。

## 8. 验收标准

- 未授权租户不能访问、查看或分配应用权限。
- 应用禁用或过期后相关权限立即失效。
- 应用恢复后历史角色绑定重新生效。
- 平台默认权限不能被租户取消。
- 租户可以撤销自己追加的权限。
- 自定义角色不能提交未授权应用权限。
- 租户管理员自动获得新开通应用和应用新增权限。
- 角色 ID 相同但来源不同不会发生错误关联。
- 现有租户切换后有效应用数量不意外减少。

## 9. 灰度策略

- 按租户启用新租户应用和角色授权模型。
- 默认租户最后切换。
- 每批租户切换后比较权限、菜单和关键 API 访问结果。

## 10. 退出条件

- 所有活跃租户已生成明确应用授权。
- 所有角色权限写入使用新权限表达。
- 租户灰度期间无非预期越权或权限缺失。

## 11. 回滚

- 按租户关闭新授权模型。
- 恢复旧应用状态继承逻辑。
- 保留新租户应用授权和角色来源数据，等待问题修复后再次切换。
