# 应用中心化授权改造 Spec

> 状态：approved（设计已确认，尚未开始实施）

## 1. 文档目的

本目录定义 PMS 菜单、应用、权限、角色和租户应用授权的目标架构及分阶段实施计划。

改造遵循以下原则：

- 应用是资源归属和租户授权边界。
- 菜单负责导航，权限负责能力，二者都归属于应用。
- 菜单权限自动创建和同步，操作权限在应用权限树中维护。
- 精确权限只匹配自身，只有显式 `:*` 权限包含后代。
- 平台预设角色共享使用，租户只能追加权限，不能削减平台默认权限。
- 前端继续接收菜单树，根菜单先按应用排序，再按菜单排序。
- 全程采用兼容迁移，不在单次发布中直接替换旧模型。

## 3. 阶段总览

| 阶段 | 主要交付 | 是否改变线上授权结果 |
|---|---|---|
| 0 | 基线测试、数据审计、双模型差异框架 | 否 |
| 1 | 新字段、索引、应用与权限数据回填 | 否 |
| 2 | 应用中心化管理接口和写入链路 | 默认否 |
| 3 | 新权限匹配器和有效权限计算 | 影子运行 |
| 4 | 新租户应用授权和角色授权模型 | 可按租户灰度 |
| 5 | 新菜单生成正式接管 | 是，受开关控制 |
| 6 | 删除旧模型 | 是，不可直接回退到旧代码 |

## 4. 实施顺序

阶段必须按顺序执行：

```text
0 → 1 → 2 → 3 → 4 → 5 → 6
```

进入下一阶段前，必须满足当前阶段的全部验收标准和退出条件。

阶段 0 至阶段 5 均需保留可回滚能力。阶段 6 只能在新模型稳定运行至少一个完整发布周期后执行。

## 5. 不在本次范围内

- 不引入完整 ABAC 或策略语言。
- 不把菜单、权限和应用合并为单表。
- 不重新设计部门数据权限模型。
- 不强制前端增加应用一级导航。
- 不为每个 HTTP 接口创建独立权限。
- 不复制平台预设角色到每个租户。
